# Plan: Multi-Step Chapters

Allow a single chapter to contain multiple exercise files (`.rs`) and
markdown notes (`.md`) in a defined order, so prose and code can
interleave. Each `.rs` file is a self-contained step with its own
exercise functions and tests.

This is the design agreed in the conversation summarised in
`docs/architecture.md`. This file is the executable plan: the steps
below are small and independently shippable. Tick them off as you go
and append notes inline.

## Target shape

```
examples/07_option/
├── 1_intro.md           # prose only
├── 2_unwrap.rs          # exercise + tests
├── 3_pattern_match.md   # prose between exercises
├── 4_combinators.rs     # exercise + tests
├── 5_hints.md           # special slug, surfaced as "Stuck?"
└── main.rs              # auto-generated by build.rs
```

Conventions:

- File ordering is by the leading `N_` prefix, mixing `.rs` and `.md`
  freely.
- `.rs` step files start with a `//!` inner doc whose first `# H1`
  becomes the step title; the rest is rendered as prose above that
  step's editor pane. Same convention as today's chapter `main.rs`.
- Each `.rs` step's tests live in a `#[cfg(test)] mod tests` block
  inside that file. Step files are self-contained: no
  `use super::<other_step>::*` between steps. Tests in step N only
  exercise items defined in step N.
- The chapter's `main.rs` is generated; contributors never edit it.
  It contains `mod _2_unwrap; mod _4_combinators; fn main() {}` plus
  a `// @generated` header.
- A chapter with only `main.rs` (single-step legacy shape) keeps
  working: `scan_dir` treats it as a one-step chapter.

## Running and testing

- Whole chapter: `cargo test --example 07_option` (unchanged).
- One step: `cargo test --example 07_option _2_unwrap::` (Cargo module
  filter). The CLI hides this detail: `cargo course submit
  examples/07_option/2_unwrap.rs` infers chapter + step from the path
  and runs the right filter.
- Online: each `Code` step renders as its own editor block with
  prose above and below; "Run" / "Test" buttons act on that step's
  source only. The server proxies a synthesised single-file program
  per step to play.rust-lang.org (just that step's source, no need
  for the chapter aggregator).

## Data model changes

`Exercise` becomes a chapter that owns an ordered `Vec<Step>`:

```rust
pub enum Step {
    Prose(Note),                   // .md file
    Code(CodeStep),                // .rs file
}

pub struct CodeStep {
    pub order: u32,
    pub slug: String,              // e.g. "2_unwrap"
    pub file_stem: String,         // e.g. "_2_unwrap"
    pub title: String,
    pub intro_html: String,        // from //! doc
    pub starter_code: String,      // file content with //! stripped
    pub hints: Option<Note>,       // per-step hints? see Step 4 below
}
```

`hints` currently lives per chapter. Decision for v1: keep
chapter-level `hints` (a sibling `N_hints.md`) for simplicity. A
future iteration can promote it to per-step if needed.

`submissions.exercise_name` becomes `<chapter>/<step>`, e.g.
`07_option/2_unwrap`. The DB is throwaway, so we wipe and re-migrate
rather than backfill.

## Step-by-step rollout

Each numbered step is intended to be a single PR/commit. Steps 1-3 are
infrastructure with no learner-visible change; step 4 is the first
pilot chapter; later steps migrate the rest and clean up.

### Step 1 - Data model + scanner, single-step compatible

- Introduce `Step` / `CodeStep` in `src/exercises.rs`.
- Rewrite `parse_chapter` to:
  - List `*.rs` and `*.md` files (excluding `main.rs` and `hints.md`).
  - For chapters with only `main.rs`, synthesise a single `Code` step
    from it (preserves current behaviour exactly).
  - For chapters with sibling `*.rs` files, parse each as a
    `CodeStep` and interleave with `Prose` steps by `N_` prefix.
- Keep the public `Exercise` fields used by templates by exposing
  convenience accessors (`fn primary_step`, `fn code_steps`, etc.)
  so the template change can land in a later step.
- Update unit tests in `exercises.rs` to cover both shapes.

Acceptance: `cargo test` green; no template or server changes.

### Step 2 - `build.rs` for chapter aggregators

- Add `build.rs` at the crate root. For each `examples/NN_<slug>/`
  that contains sibling `*.rs` files, generate `main.rs` with:
  ```
  // @generated by build.rs - do not edit
  #![allow(dead_code, unused_imports)]
  mod _2_unwrap;
  mod _4_combinators;
  fn main() {}
  ```
- Skip chapters that have no sibling `.rs` (legacy single-file
  chapters keep their hand-written `main.rs`).
- Emit `cargo:rerun-if-changed=examples` so adding/renaming step
  files regenerates.
- Add `examples/*/main.rs` (only those that are generated) to
  `.gitignore`? Decision: **commit them anyway** so `cargo test
  --example` works for users who haven't run `build.rs` (e.g.
  Playground users copying files). The `build.rs` just keeps them in
  sync; CI verifies no drift.

Acceptance: build succeeds, generated `main.rs` files match
hand-rolled equivalents byte-for-byte for a test chapter.

### Step 3 - Server + templates render steps

- Change `exercise.html` to iterate `chapter.steps` and render either
  a prose block or a `<section>` containing prose + editor + run/test
  buttons per code step.
- `ExerciseTemplate` gains a `Vec<RenderStep>` field; existing
  per-chapter fields (`starter_code`, `intro_html`) become derived
  from the first/only code step for legacy chapters.
- `dots: Vec<ProgressDot>` becomes a list over code steps across all
  chapters (so the "next" button moves to the next code step, which
  may be in the same chapter).
- Update `/api/run` and `/api/format` to accept a step identifier so
  the server proxies the right source. Request shape:
  `{ chapter: "07_option", step: "2_unwrap", code: "…" }`.
- Update `/api/submit` to accept `{ chapter, step, source, … }` and
  write `exercise_name = "<chapter>/<step>"`.

Acceptance: Existing single-step chapters render identically to
before; the URL `/exercise/07_option` still works.

### Step 4 - Pilot chapter: convert `07_option`

- Split today's `examples/07_option/main.rs` into:
  - `1_intro.md` (overview prose, currently the top of the `//!`
    block).
  - `2_<topic>.rs` ... `N_<topic>.rs` step files, one concept each.
  - Keep `N_hints.md` at the chapter level.
- Verify:
  - `cargo test --example 07_option` runs all steps.
  - `cargo test --example 07_option _2_<slug>::` runs one step.
  - Online view shows interleaved prose + multiple editors.
  - `cargo course submit examples/07_option/2_<slug>.rs` works.

Acceptance: pilot chapter passes locally and online; no regressions
in other chapters.

### Step 5 - CLI ergonomics

- `cargo course submit <path>`:
  - If path is `examples/<chapter>/<N>_<slug>.rs`, derive chapter +
    step and run `cargo test --example <chapter> _<N>_<slug>::`.
  - If path is `examples/<chapter>/main.rs` (legacy), behave as today.
  - With no path, pick the next unfinished code step from `status`.
- `cargo course status` prints one row per code step under each
  chapter, with chapter rows summarising "M of N steps passed".

Acceptance: CLI works end-to-end against a running server with the
pilot chapter.

## Open questions / decisions to revisit

- **Per-step hints.** Deferred. Revisit if learners ask for hints
  on a specific step rather than the whole chapter.
- **Module name prefix.** Files starting with a digit need a Rust-
  legal module name. Plan uses a leading underscore (`_2_unwrap`).
  Alternative: strip the number from the module name (`mod unwrap;`)
  and only use the number for ordering. Underscore is simpler and
  preserves the visible mapping; go with it unless it bothers us.
- **Generated `main.rs` in git.** Plan: commit. Revisit if churn
  becomes noisy.
- **Playground "Run" for a step.** The server already synthesises
  the snippet sent to play.rust-lang.org. For a step file that
  references `super::` items, we'd need to inline. Decision: forbid
  cross-step `use super::` by convention, so each step is a valid
  standalone program once `fn main() {}` is appended.
